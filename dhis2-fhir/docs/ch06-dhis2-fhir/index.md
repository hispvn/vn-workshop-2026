# DHIS2 and FHIR

DHIS2 (District Health Information Software 2) is an open-source platform used in more than 80 countries for health data collection, analysis, and reporting. It supports two primary data models: **aggregate data** (summarized counts and indicators reported periodically from facilities) and **tracker data** (individual-level, event-based records following a person through a health program). Tracker programs in DHIS2 use concepts like Tracked Entity Instances, Enrollments, Events, and Data Elements to capture longitudinal health data -- for example, following a patient through an immunization schedule or a tuberculosis treatment program.

Mapping DHIS2 data to FHIR matters because health systems rarely operate in isolation. A country's HIV program data in DHIS2 may need to feed into a national shared health record, a laboratory information system, or a regional disease surveillance network -- all of which increasingly speak FHIR. Without a clear mapping between DHIS2's internal data model and FHIR resources, building these integrations requires ad-hoc custom code for every connection. A well-defined FHIR Implementation Guide for DHIS2 data provides a standard, reusable contract that any system can implement against, dramatically reducing integration effort and improving data quality.

This chapter introduces the core concepts behind mapping DHIS2 to FHIR. We start by mapping the main DHIS2 objects (Tracked Entity Instances, Enrollments, Events, Data Elements, Organisation Units, and Programs) to their closest FHIR resource counterparts. Along the way, we will discuss the conceptual mismatches between the two data models and strategies for handling them. The profiles, extensions, and logical models you built in previous chapters provide the technical foundation for expressing these mappings as formal, computable artifacts in an Implementation Guide.
