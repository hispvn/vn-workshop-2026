# Project Structure

This project is organized into three main areas: the Implementation Guide source, the learning guide documentation, and exercises. Here is an overview of the directory layout:

```
dhis2-fhir/
├── Makefile                  # Build orchestration
├── Dockerfile                # Docker setup for FHIR tooling
├── ig/                       # Implementation Guide source
│   ├── sushi-config.yaml     # SUSHI / IG Publisher configuration
│   ├── input/
│   │   ├── fsh/              # FSH source files
│   │   │   ├── profiles/     # Profile definitions
│   │   │   ├── extensions/   # Extension definitions
│   │   │   ├── valuesets/    # ValueSet definitions
│   │   │   ├── codesystems/  # CodeSystem definitions
│   │   │   ├── instances/    # Example instances
│   │   │   ├── logical/      # Logical models
│   │   │   ├── mappings/     # Mapping definitions
│   │   │   └── aliases.fsh   # Alias definitions
│   │   ├── pagecontent/      # Narrative pages for the IG (Markdown)
│   │   └── images/           # Images used in the IG
│   └── output/               # Generated IG output (git-ignored)
├── book/                     # mdbook learning guide (this documentation)
│   ├── book.toml             # mdbook configuration
│   ├── SUMMARY.md            # Table of contents
│   └── ch01-fhir-overview/   # Chapter directories
│       └── ...
└── exercises/                # Hands-on exercises
    └── ...
```

## The `ig/` Directory

This is where the Implementation Guide lives. It follows the standard SUSHI project layout:

- **`sushi-config.yaml`** -- The central configuration file. It defines the IG's id, name, version, FHIR version, dependencies, and other metadata. SUSHI and the IG Publisher both read this file.

- **`input/fsh/`** -- All FSH source files go here. SUSHI recursively scans this directory for `.fsh` files, so you can organize them into subdirectories however you like. The convention in this project is to group by artifact type (profiles, extensions, etc.).

- **`input/pagecontent/`** -- Markdown files that become narrative pages in the published IG. These are the human-readable documentation pages that accompany the computable definitions.

- **`input/images/`** -- Images referenced by the pagecontent Markdown files.

- **`output/`** -- Generated by the IG Publisher. Contains the complete HTML website for the IG. This directory is git-ignored since it can be regenerated at any time.

## The `book/` Directory

This is the mdbook learning guide you are reading right now. It contains:

- **`book.toml`** -- mdbook configuration (title, authors, build settings)
- **`SUMMARY.md`** -- The table of contents that defines the book structure
- **Chapter directories** -- Each chapter has its own directory with a `index.md` (chapter intro) and additional section files

## The `exercises/` Directory

Hands-on exercises that accompany the chapters. Each exercise provides a starting point and instructions for practicing the concepts covered in the guide.

## Why This Layout?

Keeping the IG source, documentation, and exercises in separate top-level directories makes each concern easy to find and work with independently. The `ig/` directory follows SUSHI conventions so that standard FHIR tooling works without modification. The `book/` directory follows mdbook conventions. The Makefile ties everything together, so you can build any part of the project with a single command.
