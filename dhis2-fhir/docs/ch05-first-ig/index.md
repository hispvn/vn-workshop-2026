# Building Your First Implementation Guide

An Implementation Guide (IG) is a structured package that defines how FHIR should be used for a specific purpose. It contains profiles, extensions, value sets, code systems, examples, and narrative documentation -- everything a developer or system needs to correctly implement a particular use case. Think of it as a "rulebook" that takes the broad FHIR standard and narrows it down to exactly what your project requires. IGs are the primary mechanism by which communities agree on how to exchange data: national health programs, disease surveillance networks, and platforms like DHIS2 all publish IGs to describe their FHIR interfaces.

IGs exist because the base FHIR specification is intentionally broad. A Patient resource, for example, has dozens of optional fields. Without an IG, two systems might implement Patient in incompatible ways -- one including national identifiers and the other omitting them. By publishing an IG, you make your expectations explicit and machine-testable: validators can check whether a given resource conforms to your IG's rules, and implementers can generate code from your IG's structure definitions.

Two tools form the backbone of IG development. **SUSHI** (SUSHI Unshortens ShortHand Inputs) is the FSH compiler. It reads your `.fsh` files and produces FHIR JSON artifacts (StructureDefinitions, ValueSets, CodeSystems, examples, and so on) in the `fsh-generated/` directory. The **IG Publisher** then takes those generated artifacts, validates them against the FHIR specification, and produces a complete HTML website that you can host or share. In this project, both tools run inside a Docker container, so you do not need to install Java or Node.js locally.
