# IPS Profiles

This chapter documents the four profiles that together form a DHIS2-compatible International Patient Summary. Each profile extends either a base FHIR resource or the existing DHIS2 profiles from this IG.

## Profile Overview

| Profile | Parent | Purpose |
|---------|--------|---------|
| `DHIS2IPSPatient` | DHIS2Patient | Patient demographics for IPS |
| `DHIS2IPSImmunization` | Immunization | A single vaccine administration |
| `DHIS2IPSComposition` | Composition | Document table of contents |
| `DHIS2IPSBundle` | Bundle | The complete IPS document |

## DHIS2IPSPatient

Extends the base `DHIS2Patient` profile with IPS-relevant constraints. Since `DHIS2Patient` already requires name, gender, birthDate, and a DHIS2 UID -- all of which IPS demands -- the IPS patient profile adds only a few additional Must Support elements for patient matching across systems.

### DHIS2 Context

In DHIS2 immunization programs, patients are Tracked Entity Instances (TEIs) with attributes for demographics, contact information, and identifiers. The IPS patient needs enough of this data for a clinician in another country to confidently identify the patient.

### Constraints

| Element | Cardinality | Notes |
|---------|-------------|-------|
| `identifier` | 1..* MS | Inherited: DHIS2 UID required |
| `identifier[dhis2uid]` | 1..1 MS | Inherited: 11-char DHIS2 UID |
| `identifier[national]` | 0..1 MS | Inherited: Government-issued ID |
| `name` | 1..* MS | Inherited: family + given required |
| `gender` | 1..1 MS | Inherited |
| `birthDate` | 1..1 MS | Inherited |
| `address` | 0..* MS | Inherited |
| `telecom` | 0..* MS | **New**: phone/email for patient matching |
| `maritalStatus` | 0..1 MS | **New**: IPS recommends for demographics |
| `communication.language` | 0..* MS | **New**: Patient's language(s) |

### FSH Source

```fsh
Profile: DHIS2IPSPatient
Parent: DHIS2Patient
Id: dhis2-ips-patient
Title: "DHIS2 IPS Patient"

* telecom MS
* maritalStatus MS
* communication MS
* communication.language MS
```

The profile is intentionally minimal because `DHIS2Patient` already covers IPS requirements. The additions (`telecom`, `maritalStatus`, `communication.language`) help receiving systems match patients and communicate in the right language.

### Mapping from DHIS2 Tracked Entity Attributes

| DHIS2 TEA | FHIR Element | Notes |
|-----------|--------------|-------|
| First name | `name[0].given[0]` | |
| Last name | `name[0].family` | |
| Sex | `gender` | Map DHIS2 option set to `male`/`female`/`other`/`unknown` |
| Date of birth | `birthDate` | |
| Mobile | `telecom[0].value` | Set `system = #phone` |
| Email | `telecom[1].value` | Set `system = #email` |
| National ID | `identifier[national].value` | |
| DHIS2 UID | `identifier[dhis2uid].value` | Auto-generated by DHIS2 |
| Address (city) | `address[0].city` | |
| Address (country) | `address[0].country` | Use ISO 3166-1 alpha-2 |


## DHIS2IPSImmunization

Records a single vaccine administration event. In DHIS2, immunizations are typically captured as tracker program events (program stage entries) with data elements for vaccine type, dose number, date given, lot number, and administration site.

### DHIS2 Context

A typical DHIS2 immunization program has a program stage like "Immunization Visit" with data elements:

| DHIS2 Data Element | FHIR Immunization Element | Type |
|-------------------|--------------------------|------|
| Vaccine given | `vaccineCode` | Option set → CVX code |
| Date given | `occurrenceDateTime` | DATE |
| Dose number | `protocolApplied[0].doseNumberPositiveInt` | INTEGER |
| Lot number | `lotNumber` | TEXT |
| Injection site | `site` | Option set → SNOMED CT |
| Route | `route` | Option set → SNOMED CT |
| Disease | `protocolApplied[0].targetDisease` | Derived from vaccine |
| Facility | `performer[0].actor` | Organisation unit |
| Event status | `status` | COMPLETED → `completed` |

### Constraints

| Element | Cardinality | Notes |
|---------|-------------|-------|
| `status` | 1..1 MS | `completed`, `entered-in-error`, or `not-done` |
| `vaccineCode` | 1..1 MS | Must include at least one coding |
| `vaccineCode.coding[cvx]` | 0..1 MS | CDC CVX code (system: `http://hl7.org/fhir/sid/cvx`) |
| `vaccineCode.coding[whoAtc]` | 0..1 MS | WHO ATC code (system: `http://www.whocc.no/atc`) |
| `patient` | 1..1 MS | Reference to the IPS Patient |
| `occurrence[x]` | 1..1 MS | Date/time of administration |
| `lotNumber` | 0..1 MS | Vaccine lot/batch number |
| `site` | 0..1 MS | Body site (SNOMED CT) |
| `route` | 0..1 MS | Route of administration (SNOMED CT) |
| `performer` | 0..* MS | Who administered the vaccine |
| `protocolApplied` | 0..* MS | Dose protocol |
| `protocolApplied.doseNumber[x]` | 0..1 MS | Dose number in series |
| `protocolApplied.targetDisease` | 0..* MS | Disease the vaccine protects against |

### Vaccine Code Slicing

The `vaccineCode.coding` element is sliced to support two international code systems simultaneously:

```fsh
* vaccineCode.coding ^slicing.discriminator.type = #pattern
* vaccineCode.coding ^slicing.discriminator.path = "system"
* vaccineCode.coding ^slicing.rules = #open

* vaccineCode.coding contains
    cvx 0..1 MS and
    whoAtc 0..1 MS
```

This means a single immunization can carry **both** a CVX code and a WHO ATC code:

```json
"vaccineCode": {
  "coding": [
    {
      "system": "http://hl7.org/fhir/sid/cvx",
      "code": "19",
      "display": "BCG"
    },
    {
      "system": "http://www.whocc.no/atc",
      "code": "J07AN01",
      "display": "BCG vaccine"
    }
  ]
}
```

Both slices are 0..1 (optional) because some implementations only have one code system available. The slicing rules are `#open`, allowing additional code systems (like a DHIS2 option set code) alongside the standard ones.

### Dual-Coding Pattern for DHIS2

When exporting from DHIS2, the recommended pattern is to include both the standard code and the DHIS2 option set code:

```json
"vaccineCode": {
  "coding": [
    {
      "system": "http://hl7.org/fhir/sid/cvx",
      "code": "19",
      "display": "BCG"
    },
    {
      "system": "http://dhis2.org/fhir/id/option-set/vaccine-type",
      "code": "BCG",
      "display": "BCG"
    }
  ]
}
```

The standard code (CVX) ensures interoperability with any IPS-aware system. The DHIS2 code enables lossless round-trip back to DHIS2. This dual-coding pattern is possible because the slicing rules are `#open`.

### FSH Source

```fsh
Profile: DHIS2IPSImmunization
Parent: Immunization
Id: dhis2-ips-immunization
Title: "DHIS2 IPS Immunization"

* status MS
* vaccineCode 1..1 MS
* vaccineCode.coding 1..* MS

* vaccineCode.coding ^slicing.discriminator.type = #pattern
* vaccineCode.coding ^slicing.discriminator.path = "system"
* vaccineCode.coding ^slicing.rules = #open

* vaccineCode.coding contains
    cvx 0..1 MS and
    whoAtc 0..1 MS

* vaccineCode.coding[cvx].system = "http://hl7.org/fhir/sid/cvx" (exactly)
* vaccineCode.coding[cvx].code 1..1
* vaccineCode.coding[cvx].display 1..1

* vaccineCode.coding[whoAtc].system = "http://www.whocc.no/atc" (exactly)
* vaccineCode.coding[whoAtc].code 1..1

* patient 1..1 MS
* occurrence[x] 1..1 MS
* lotNumber MS
* site MS
* route MS
* performer MS
* protocolApplied MS
* protocolApplied.doseNumber[x] MS
* protocolApplied.targetDisease MS
```


## DHIS2IPSComposition

The Composition is the **table of contents** of the IPS document. It does not contain clinical data itself -- it organizes references to the Patient and Immunization resources into named sections.

### IPS Required Sections

The HL7 IPS specification requires three sections (Allergies, Problems, Medications). Our DHIS2 IPS profile focuses on immunization data, so it requires only the **Immunizations** section. Sections without data can use `emptyReason` in a full IPS implementation.

### Constraints

| Element | Cardinality | Notes |
|---------|-------------|-------|
| `status` | 1..1 MS | Typically `final` |
| `type` | 1..1 MS | Fixed: LOINC `60591-5` "Patient summary Document" |
| `subject` | 1..1 MS | Reference to DHIS2IPSPatient |
| `date` | 1..1 MS | When the summary was generated |
| `author` | 1..* MS | Who/what created it (can be a system) |
| `title` | 1..1 MS | Fixed: "International Patient Summary" |
| `section[immunizations]` | 1..1 MS | Required immunizations section |
| `section[immunizations].code` | 1..1 | LOINC `11369-6` "History of Immunization Narrative" |
| `section[immunizations].entry` | 0..* MS | References to Immunization resources |

### FSH Source

```fsh
Profile: DHIS2IPSComposition
Parent: Composition
Id: dhis2-ips-composition
Title: "DHIS2 IPS Composition"

* status MS
* type 1..1 MS
* type = http://loinc.org#60591-5 "Patient summary Document"
* subject 1..1 MS
* subject only Reference(DHIS2IPSPatient)
* date 1..1 MS
* author 1..* MS
* title 1..1 MS
* title = "International Patient Summary"

* section 1..* MS
* section ^slicing.discriminator.type = #pattern
* section ^slicing.discriminator.path = "code"
* section ^slicing.rules = #open

* section contains immunizations 1..1 MS

* section[immunizations].title = "Immunizations"
* section[immunizations].code 1..1
* section[immunizations].code = http://loinc.org#11369-6
* section[immunizations].entry MS
* section[immunizations].entry only Reference(DHIS2IPSImmunization)
```

### Section Slicing Explained

The Composition's `section` element is sliced by `code`, which means each section is identified by its LOINC code. The immunizations section uses LOINC `11369-6` ("History of Immunization Narrative"). The slicing rules are `#open`, so additional sections (allergies, medications, etc.) can be added without changing the profile.


## DHIS2IPSBundle

The Bundle is the **transport wrapper** -- the actual JSON document that gets sent between systems. It wraps the Composition, Patient, and all Immunization resources into a single payload.

### Constraints

| Element | Cardinality | Notes |
|---------|-------------|-------|
| `type` | 1..1 | Fixed: `document` |
| `timestamp` | 1..1 MS | When the bundle was assembled |
| `entry` | 1..* MS | All resources in the document |
| `entry.resource` | 1..1 MS | The actual resource |

### Entry Order

The first entry **must** be the Composition. After that, the Patient and Immunization resources follow in any order:

```
entry[0]: Composition   ← always first (table of contents)
entry[1]: Patient       ← the subject
entry[2]: Immunization  ← first vaccine record
entry[3]: Immunization  ← second vaccine record
...
```

### FSH Source

```fsh
Profile: DHIS2IPSBundle
Parent: Bundle
Id: dhis2-ips-bundle
Title: "DHIS2 IPS Bundle"

* type = #document (exactly)
* timestamp 1..1 MS
* entry 1..* MS
* entry.resource 1..1 MS
```

### Self-Contained Document

Every resource referenced from the Composition must be included as an entry in the Bundle. The Bundle is designed to be **self-contained** -- a receiving system should be able to process it without any external lookups (except for standard terminology like LOINC or SNOMED CT, which are assumed to be known).


## How the Profiles Work Together

Here is how a complete IPS document flows from DHIS2 data to a FHIR Bundle:

```
DHIS2 Tracker Data                    FHIR IPS Bundle
─────────────────                    ─────────────────
TEI (Person)           ──────────►   Patient
  ├─ UID: DXz2k5eGbri                 ├─ identifier[dhis2uid]
  ├─ Name: Jane Doe                    ├─ name: Jane Doe
  ├─ Sex: Female                       ├─ gender: female
  ├─ DOB: 1990-05-15                   ├─ birthDate: 1990-05-15
  └─ Phone: +265999...                 └─ telecom: +265999...

Event (Immunization Visit) ────────►   Immunization
  ├─ Vaccine: BCG                      ├─ vaccineCode: CVX#19
  ├─ Date: 2019-01-15                  ├─ occurrenceDateTime
  ├─ Dose: 1                           ├─ protocolApplied.doseNumber
  ├─ Lot: AB1234                       ├─ lotNumber
  └─ Site: Left arm                    └─ site: SNOMED#368208006

                                     Composition
                                       ├─ type: LOINC#60591-5
                                       ├─ subject → Patient
                                       └─ section[immunizations]
                                            └─ entry → Immunization

                                     Bundle (type: document)
                                       ├─ entry[0]: Composition
                                       ├─ entry[1]: Patient
                                       └─ entry[2]: Immunization
```
