# Local Resource Renderer

Throughout this book we have been writing FSH profiles, extensions, value sets, and instances. SUSHI compiles these into FHIR JSON files, and the IG Publisher turns them into a static documentation site. But the IG Publisher output is oriented toward implementers reading specifications -- it shows structure definitions, differential tables, and narrative summaries. It does not show you what a Patient *looks like* as a profile card, or what a Questionnaire *feels like* as a fillable form. That is what the local renderer provides.

The renderer is a small Python web application built on FastAPI. It reads the same JSON files that SUSHI generates, parses them into Pydantic models, and renders them as HTML pages you can view in your browser. It supports three resource types:

- **Patients** are rendered as profile cards showing demographics, identifiers, address, and links to any QuestionnaireResponses submitted for that patient.
- **Questionnaires** are rendered as interactive HTML forms. Each item becomes the appropriate input type -- text fields, date pickers, dropdowns, radio buttons -- based on the FHIR `item.type`. Answer options are resolved through the ValueSet and CodeSystem chain.
- **QuestionnaireResponses** are rendered as pre-filled forms. The renderer finds the matching Questionnaire definition, extracts the stored answers, and populates the same form layout with the response data.

This gives you a fast feedback loop while authoring FSH. Edit a `.fsh` file, run `make sushi` to regenerate JSON, and refresh the browser to see the result rendered as a real application would display it.

## Running the renderer

The renderer runs in development mode with auto-reload:

```bash
make run
```

This calls:

```bash
uv run fastapi dev src/dhis2_fhir/app.py
```

Once running, open `http://127.0.0.1:8000` in your browser. The index page lists all Patients, Questionnaires, and QuestionnaireResponses found in the SUSHI output directories.

FastAPI automatically generates OpenAPI documentation at `http://127.0.0.1:8000/docs`, which can be useful for understanding the available routes and their parameters.

## How it works

The renderer has three layers:

1. **Models** (`models.py`) -- Pydantic models that mirror FHIR resource structures. These handle parsing JSON into typed Python objects with computed properties for display logic. See [Pydantic Models](pydantic-models.md) for the full reference.

2. **Loader** (`loader.py`) -- Functions that scan the SUSHI output directories for JSON files, deduplicate by resource id, and parse them into model instances. The loader also indexes CodeSystems and ValueSets by URL for fast option resolution.

3. **App** (`app.py`) -- FastAPI routes that load resources, resolve answer options, and render Jinja2 templates. See [Architecture](architecture.md) for the full breakdown.

## Design philosophy

The models are intentionally permissive. Most fields default to empty strings or empty lists rather than being required. This is a deliberate choice: we are parsing *instance* data generated by SUSHI, not validating conformance. A Patient example might omit `gender` or `birthDate` entirely, and the renderer should still work. The Pydantic models use `Field(default_factory=list)` for list fields and empty-string defaults for strings, so any valid FHIR JSON will parse without errors.

Computed properties like `Patient.display_name` and `HumanName.display` centralize display formatting. Templates call these properties rather than assembling strings inline, which keeps the Jinja2 templates clean and the logic testable.
